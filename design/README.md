## 你直接改这个文件：`design/moves.json`

引擎只读这一个文件。你改它 = 改产品内容，立即生效，不需要我改代码。

### 1. `moves` —— 抽卡动作库（核心）

每个动作一个对象，字段如下：

| 字段 | 含义 | 例 | 备注 |
|---|---|---|---|
| `zh` | 动作中文名（卡片大标题） | `"深蹲"` | 必填 |
| `en` | 英文名（标题下小字） | `"Squat"` | 必填 |
| `emoji` | 卡片中央大图标 | `"🏋️"` | 先用 emoji 占位；之后可换成插画，换法我来接 |
| `goal` | 属于哪个目标 | `"strength"` | **必须是下面 `goals` 里的 key 之一** |
| `part` | 锻炼部位（卡片小标签） | `"下肢"` | 必填，文字自由 |
| `desc` | 动作说明（卡片正文） | `"脚与肩同宽…"` | 必填，建议 30~60 字，别太长卡片放不下 |

> 想加动作 = 往数组里加一个对象。想删 = 删掉。顺序无所谓（抽卡是随机的）。

### 2. `goals` —— 健身目标分类

`key: 显示名`。key 是给代码用的英文标识，显示名是给用户看的中文。
```json
"goals": { "lose": "减脂", "strength": "力量", "stretch": "拉伸" }
```
- 想增减目标分类，改这里。
- ⚠️ 改 key 的话，`moves` 里所有用到这个 key 的 `goal` 也要同步改，否则那个动作抽不到。
- 抽卡是**加权随机**：命中当前目标的动作出现概率 ×3，其它 ×1（这条规则在引擎里，想调权重告诉我）。

### 3. `config` —— 文案和行为

| 字段 | 含义 | 默认 |
|---|---|---|
| `defaultGoal` | 默认健身目标 | `"strength"` |
| `durationSec` | 单次运动时长（秒） | `300`（5分钟）。**demo 时我会临时改 10** |
| `deckHint` | 牌背上的提示文字 | `"抽 一 张 · 动 5 分 钟"` |
| `timerSub` | 倒计时下方鼓励语 | `"动起来…"` |
| `doneText` | 完成时的文案 | `"完成 ✓ …"` |

文案都归你，随便改。

## 卡片的视觉/交互结构（现状，供你设计参照）

当前流程是：**牌背 → 点「抽一张」→ 翻面出动作 → 点「开始 5 分钟」→ 环形倒计时 → 完成 → 螃蟹比心**。

卡片正面现在的元素，从上到下：
```
[目标标签]  ← goal 的显示名，朱砂色小字
深蹲        ← zh，大标题
Squat       ← en，灰色小字
🏋️          ← emoji，居中大图标
脚与肩同宽…  ← desc，正文
[力量][下肢][5分钟]  ← 三个小标签
```
视觉是「牛皮纸 + 手绘卡牌感」（延续 Spark Garden 风）。
**如果你想改布局/加元素/换交互（比如抽 3 张选 1、加难度选择），写在下面的 `interaction-spec.md`，我们对一下再动结构。** 改 json 内容不用对，改结构要对。

## ⚠️ 一个待定的产品决策：要不要记录健身数据？

你提到「接受/拒绝是否记录用户健身数据」。这是个**产品决策，不是这次要急着实现的**。
现在引擎完成后只做两件事：关掉浮窗 + 让螃蟹比心。**没有任何数据记录**——这是故意的，因为：

1. 一旦记录数据，就涉及「存哪、隐私、用户要不要同意」，是另一个量级的工程；
2. 核心假设还没验证：「等 agent 时被提醒做微健身，人真的会动吗」。这个用不到数据记录就能验证。

**建议：先把「抽卡→运动→完成」跑通，确认人会动，再决定记不记录。** 如果你设计方案里需要它，写在 `interaction-spec.md` 的「待定决策」一节，我们一起想最小实现（比如只在本地存个连续天数，不碰隐私）。
